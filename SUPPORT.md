# Support

This is the default support policy for public repositories owned by [@prprnya](https://github.com/prprnya). Individual repositories may provide their own `SUPPORT.md`; when they do, the repository-specific support policy takes precedence.

## Where to ask for help

Use the issue tracker of the relevant repository for questions that are specific to that repository, including:

- usage questions
- documentation gaps
- reproducible bugs
- feature requests
- maintenance suggestions
- compatibility questions

Before opening a new issue, please check the repository's README, documentation, existing issues, and pinned issues when available.

## Bug reports

For reproducible bugs, open a bug report in the affected repository and include:

- what happened
- what you expected to happen
- reproduction steps
- relevant versions, environment details, logs, screenshots, or links

Use the repository's issue form if one is available.

## Feature requests

For feature requests, describe the problem or motivation first. A good request explains the use case, expected benefit, possible alternatives, and any compatibility concerns.

Large or architectural proposals should start as an issue before a pull request.

## Pull requests

For contribution guidance, see [`CONTRIBUTING.md`](CONTRIBUTING.md). Repository-specific contribution instructions take precedence when present.

## Security vulnerabilities

Do not report security vulnerabilities through public issues, pull requests, discussions, or comments.

Report potential vulnerabilities privately by following [`SECURITY.md`](SECURITY.md).

## Code of Conduct reports

Do not report harassment, abuse, or other Code of Conduct concerns through public issues, pull requests, discussions, or comments.

Report possible violations privately by following [`CODE_OF_CONDUCT.md`](CODE_OF_CONDUCT.md).

## Response expectations

These are personal projects and open source repositories maintained as availability allows. Opening an issue or pull request does not guarantee a response, implementation, review, merge, release, or support window.

Maintainer decisions may prioritize correctness, maintainability, security, scope control, and long-term project direction over immediate support requests.
